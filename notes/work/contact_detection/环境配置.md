# GEAR-SONIC 最小 Demo 环境配置手册

本文档用于在桌面工作站上运行 NVIDIA GEAR-SONIC 官方最小 Demo：

```text
MuJoCo Python 仿真
        ↕ Unitree SDK2 / DDS
SONIC C++ TensorRT 推理
        + 官方 ONNX 权重
```

本阶段不安装 Isaac Lab，不训练 SONIC，也不要求 ROS 2。ROS 2 工作空间
`~/g1_contact_ws` 与上游 SONIC 仓库保持分离。

## 1. 目标和非目标

完成后应能做到：

1. 打开 MuJoCo G1 仿真窗口；
2. 启动 SONIC C++ 推理程序；
3. C++ 推理程序与 MuJoCo 通过 Unitree SDK2/DDS 通信；
4. G1 在仿真中落地、保持平衡并执行官方参考动作。

当前不做：

- Isaac Lab / Isaac Sim；
- SONIC 训练或微调；
- ROS 2 接口；
- 黑板和接触状态算法；
- 真机控制。

## 2. 推荐版本

| 组件 | 推荐/要求 |
| --- | --- |
| 系统 | Ubuntu 22.04 x86_64 |
| Python（MuJoCo） | 官方脚本管理的 Python 3.10 |
| CUDA | CUDA Toolkit 12.x；建议 12.4 |
| TensorRT（桌面 x86_64） | 10.13，必须匹配官方要求 |
| Git | 支持 Git LFS |
| GPU | 支持 CUDA/TensorRT 的 NVIDIA GPU |
| 显示 | 原生 X11/Wayland，或配置正确的 WSLg |

GEAR-SONIC 官方部署文档要求桌面 x86_64 使用 TensorRT 10.13。不要用
Jetson/G1 Orin 所需的 TensorRT 10.7 替代。

## 3. 目录规划

建议保持以下结构：

```text
~/
├── g1_contact_ws/                 # 自己的 ROS 2 工程，暂时不参与 Demo
│   ├── docs/
│   └── src/
└── g1_sonic/
    └── GR00T-WholeBodyControl/    # NVIDIA 官方仓库
```

不要把 `GR00T-WholeBodyControl` 放进 `g1_contact_ws/src`，它不是普通的
ROS 2 package。

## 4. WSL2 用户必须先检查 GPU

当前开发环境是 Ubuntu 22.04 WSL2。如果使用原生 Ubuntu，可跳到第 5 节。

### 4.1 Windows 侧

在管理员 PowerShell 中执行：

```powershell
wsl --update
wsl --shutdown
```

安装最新 NVIDIA Windows 驱动，然后重新进入 WSL。

WSL 只安装 Windows 侧 NVIDIA 驱动。不要在 WSL 内安装
`nvidia-driver-*` 或 Linux 显卡驱动。

### 4.2 WSL 内验证 GPU 透传

```bash
ls -l /dev/dxg
```

```bash
/usr/lib/wsl/lib/nvidia-smi
```

可选地把 WSL NVIDIA 工具加入当前终端：

```bash
export PATH=/usr/lib/wsl/lib:$PATH
```

```bash
nvidia-smi
```

必须满足：

- `/dev/dxg` 存在；
- `nvidia-smi` 能识别 GPU；
- 没有 `GPU access blocked` 或驱动通信失败。

若这一步失败，先修复 Windows 驱动和 WSL GPU 透传，不要继续安装
TensorRT。WSL 中 `nvidia-smi` 的功能可能少于原生 Linux，但必须至少能识别
GPU。

```

```

## 6. 安装 CUDA Toolkit

### 6.1 WSL2

进入 NVIDIA CUDA 下载页，选择：

```text
Linux → x86_64 → WSL-Ubuntu → deb
```

按照页面生成的命令添加 NVIDIA CUDA 软件源，然后只安装 Toolkit，例如：

```bash
sudo apt install cuda-toolkit-12-4


# 或者直接用命令行，下面安装CUDA Toolkit 12.4
cd /tmp
wget https://developer.download.nvidia.com/compute/cuda/repos/wsl-ubuntu/x86_64/cuda-keyring_1.1-1_all.deb
sudo dpkg -i cuda-keyring_1.1-1_all.deb
sudo apt update

sudo apt install cuda-toolkit-12-4
```

WSL 中不要安装这些元包：

```text
cuda
cuda-12-x
cuda-drivers
nvidia-driver-xxx
```

它们可能尝试在 WSL 内安装 Linux NVIDIA 驱动。

### 6.2 原生 Ubuntu

原生 Ubuntu 可以按照 NVIDIA CUDA Toolkit 官方安装页安装 CUDA 12.x。
显卡驱动应先通过 `nvidia-smi` 验证。

### 6.3 配置当前终端

如果安装的是 CUDA 12.4：

```bash
export CUDA_HOME=/usr/local/cuda-12.4
export CUDAToolkit_ROOT=/usr/local/cuda-12.4
export PATH=$CUDA_HOME/bin:$PATH
export LD_LIBRARY_PATH=$CUDA_HOME/lib64:${LD_LIBRARY_PATH:-}
```

检查：

```bash
nvcc --version
nvidia-smi
```

`nvidia-smi` 显示的是驱动支持的 CUDA 上限；`nvcc --version` 才表示本机安装的
CUDA Toolkit 版本。

确认正确后，可将以上三个 `export` 加到 `~/.bashrc`。

## 7. 安装 TensorRT 10.13

从 NVIDIA Developer 下载桌面 Linux x86_64、CUDA 12.x 对应的
TensorRT 10.13 TAR 包。建议使用 TAR，不使用 DEB。

创建安装目录：

```bash
mkdir -p ~/opt
```

解压下载的 TAR 包。下面的文件名需要替换为实际文件名：

```bash
tar -xf ~/Downloads/TensorRT-10.13.x.x.Linux.x86_64-gnu.cuda-12.x.tar.gz \
  -C ~/opt
```

建立稳定路径。右侧目录名替换为实际解压目录：

```bash
ln -sfn ~/opt/TensorRT-10.13.x.x ~/TensorRT
```

配置当前终端：

```bash
export TensorRT_ROOT=$HOME/TensorRT
export PATH=$TensorRT_ROOT/bin:$PATH
export LD_LIBRARY_PATH=$TensorRT_ROOT/lib:${LD_LIBRARY_PATH:-}
```

有些 TAR 包使用 `lib64`，如果 `$TensorRT_ROOT/lib` 不存在，应把上面的
`lib` 改成 `lib64`。

检查：

```bash
ls $TensorRT_ROOT/include/NvInfer.h
```

```bash
find $TensorRT_ROOT -maxdepth 2 -name 'libnvinfer.so*'
```

确认正确后，可将 `TensorRT_ROOT`、`PATH` 和 `LD_LIBRARY_PATH` 设置加入
`~/.bashrc`。

## 8. 克隆 GEAR-SONIC 官方仓库

先确保没有激活 Conda 环境：

```bash
conda deactivate
```

如果终端提示 `conda: command not found`，可以忽略。

创建目录并克隆：

```bash
mkdir -p ~/g1_sonic
cd ~/g1_sonic
```

```bash
git clone https://github.com/NVlabs/GR00T-WholeBodyControl.git
cd GR00T-WholeBodyControl
```

拉取大文件：

```bash
git lfs pull
```

检查：

```bash
git lfs status
```

如果网格、模型或二进制文件只有几十到几百字节，通常说明 Git LFS 没有正确
拉取，应先解决 LFS 问题。

## 9. 创建 MuJoCo Python 环境

从仓库根目录执行：

```bash
cd ~/g1_sonic/GR00T-WholeBodyControl
```

```bash
bash install_scripts/install_mujoco_sim.sh
```

官方脚本会：

1. 自动安装 `uv`；
2. 安装由 `uv` 管理的 Python 3.10；
3. 创建仓库内 `.venv_sim`；
4. 安装 `gear_sonic[sim]`；
5. 安装仓库内的 `unitree_sdk2_python`。

注意：该脚本会删除旧的 `.venv_sim` 后重建。后续如果已经在环境中加入自定义
包，不要随意重复运行此脚本。

激活：

```bash
source ~/.local/bin/env
source .venv_sim/bin/activate
```

检查 Python：

```bash
which python
python --version
```

检查关键包：

```bash
python -c "import mujoco; print('MuJoCo', mujoco.__version__)"
```

```bash
python -c "import pinocchio; print('Pinocchio OK')"
```

```bash
python -c "from unitree_sdk2py.core.channel import ChannelFactoryInitialize; print('Unitree SDK2 OK')"
```

不要在激活 Conda 环境后再激活 `.venv_sim`，也不要把 Isaac Lab 安装进
`.venv_sim`。

## 10. 下载官方 SONIC 部署权重

保持 `.venv_sim` 激活并位于仓库根目录：

```bash
uv pip install huggingface_hub
```

下载默认部署 ONNX 权重和 planner：

```bash
python download_from_hf.py
```

检查：

```bash
ls -lh gear_sonic_deploy/policy/release
```

```bash
ls -lh gear_sonic_deploy/planner/target_vel/V2
```

至少应看到：

```text
gear_sonic_deploy/policy/release/model_encoder.onnx
gear_sonic_deploy/policy/release/model_decoder.onnx
gear_sonic_deploy/policy/release/observation_config.yaml
gear_sonic_deploy/planner/target_vel/V2/planner_sonic.onnx
```

当前不要运行 `python download_from_hf.py --training`，否则可能下载训练 checkpoint
和约 30 GB 的 SMPL 数据。

如需低延迟部署版本，可额外执行：

```bash
python download_from_hf.py --low-latency
```

## 11. 配置并编译 C++ SONIC 推理程序

退出 Python venv：

```bash
deactivate
```

确认当前没有 Conda 环境：

```bash
conda deactivate
```

进入部署目录：

```bash
cd ~/g1_sonic/GR00T-WholeBodyControl/gear_sonic_deploy
```

检查关键环境变量：

```bash
echo $CUDA_HOME
echo $TensorRT_ROOT
nvcc --version
```

只有以上检查正确后，才运行依赖脚本：

```bash
chmod +x scripts/install_deps.sh
```

```bash
./scripts/install_deps.sh
```

脚本会安装 C++ 依赖、ONNX Runtime、ZeroMQ、JSON/YAML 库、Git LFS 和
`just`。在 WSL 中应事先安装好 CUDA Toolkit，避免脚本因为找不到 CUDA 而尝试
安装不合适的 CUDA 元包。

加载部署环境：

```bash
source scripts/setup_env.sh
```

编译：

```bash
just build
```

检查：

```bash
just --version
```

```bash
cd ..
python3 check_environment.py
```

如果 `check_environment.py` 需要 Python 包，可在 `.venv_sim` 中运行检查，但
C++ 编译和运行本身应保持 Conda 关闭。

## 12. 运行官方 Demo

使用两个全新的终端。最小 Demo 不需要 `source /opt/ros/humble/setup.bash`。

### 终端 1：MuJoCo 仿真

```bash
cd ~/g1_sonic/GR00T-WholeBodyControl
# source ~/.local/bin/env
source .venv_sim/bin/activate
python gear_sonic/scripts/run_sim_loop.py
```

预期结果：打开 MuJoCo Viewer，显示 Unitree G1。

### 终端 2：C++ SONIC 推理

```bash
cd ~/g1_sonic/GR00T-WholeBodyControl/gear_sonic_deploy
source scripts/setup_env.sh
bash deploy.sh sim
# Extra Args:        --disable-crc-check
```

### 操作顺序

1. 在终端 2 按 `]`，启动策略；
2. 点击 MuJoCo Viewer；
3. 在 Viewer 按 `9`，让 G1 落地；
4. 回到终端 2 按 `T`，执行当前参考动作；
5. 按 `N` / `P` 切换下一条/上一条动作；
6. 按 `R` 重置当前动作；
7. 按 `O` 停止控制并退出。

如果使用低延迟权重：

```bash
bash deploy.sh \
  --cp policy/low_latency/model \
  --obs-config policy/low_latency/observation_config.yaml \
  sim
```

## 14. 常见问题

### 14.1 `nvidia-smi` 失败或 `/dev/dxg` 不存在

这是 WSL GPU 透传问题，不是 Python 问题。更新 Windows NVIDIA 驱动和 WSL，
执行 `wsl --shutdown` 后重试。不要在 WSL 中安装 Linux NVIDIA 驱动。

### 14.2 `nvcc: command not found`

CUDA Toolkit 未安装，或 `$CUDA_HOME/bin` 未加入 PATH。检查：

```bash
ls /usr/local/cuda*/bin/nvcc
```

### 14.3 `NvInfer.h` 或 `libnvinfer.so` 找不到

检查：

```bash
echo $TensorRT_ROOT
find $TensorRT_ROOT -name NvInfer.h
find $TensorRT_ROOT -name 'libnvinfer.so*'
```

确认 `source scripts/setup_env.sh` 已执行，并确认 TAR 包的库目录是 `lib` 还是
`lib64`。

### 14.4 ONNX、mesh 文件很小或无法解析

通常是 Git LFS 文件仍是指针：

```bash
git lfs pull
git lfs status
```

### 14.5 MuJoCo Viewer 无法打开

检查：

```bash
echo $DISPLAY
echo $WAYLAND_DISPLAY
glxinfo -B
```

WSL 用户需确认 WSLg 工作正常。若图形与 CUDA/TensorRT 同时出现难以排查的
兼容问题，优先改用原生 Ubuntu 22.04。

### 14.6 `ChannelFactory create domain error` 或 DDS 无法连接

先确保 MuJoCo 和 C++ 部署程序运行在同一系统/WSL 实例中，并使用全新终端。
最小 Demo 不需要 ROS 2，避免遗留的 `CYCLONEDDS_URI`、`ROS_DOMAIN_ID` 或
`RMW_IMPLEMENTATION` 指向错误网卡。可以在新终端检查：

```bash
env | grep -E 'CYCLONEDDS|ROS_DOMAIN|RMW_IMPLEMENTATION'
```

若设置过自定义 DDS 网卡配置，临时取消后再按官方默认流程运行。

### 14.7 编译成功但机器人动作异常

首先核对 TensorRT 版本。桌面端必须使用官方要求的 10.13，并确保 ONNX 权重、
`observation_config.yaml` 来自同一发布版本。不要混用默认模型和低延迟模型的配置。

### 14.8 Conda 导致动态库冲突

构建和运行 C++ SONIC 前执行：

```bash
conda deactivate
```

不要把 Conda 的 CUDA、TensorRT、`libstdc++` 与系统部署环境混用。

## 15. Demo 跑通后的下一步

Demo 验收完成后再开始接触任务：

1. 在 MuJoCo 场景中增加固定黑板 geom；
2. 设计手掌接近黑板的参考动作或上肢目标；
3. 从 MuJoCo 读取手臂关节 `q`、`dq`、执行器力矩；
4. 读取 MuJoCo 接触法向力，仅作为训练标签和评测真值；
5. 接触检测算法只使用 `q`、`dq` 和力矩；
6. 仿真验证后再接入 `~/g1_contact_ws` 的 ROS 2 节点。
