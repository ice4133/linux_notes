# GEAR-SONIC G1 真机环境配置手册（aarch64 / Ubuntu 22.04）

本文档由桌面端的 `环境配置.md` 调整而来，目标是在 **Unitree G1 板载
NVIDIA Jetson Orin（aarch64）** 上原生编译并运行 SONIC C++ 推理程序，直接
通过 Unitree SDK2 / DDS 控制真机。

```text
G1 板载 Orin（Ubuntu 22.04 / aarch64）
        ├── JetPack 6 / CUDA 12.6
        ├── TensorRT 10.7（ARM64）
        ├── SONIC C++ 推理
        └── Unitree SDK2 / DDS
                    ↕
              G1 电机与传感器
```

本文档不安装 Isaac Lab，不训练 SONIC，不启动 MuJoCo，也不要求 ROS 2。

> **适用范围：**下文默认“真机”是 G1 自带的 Jetson Orin NX。仅仅满足
> `uname -m` 为 `aarch64` 并不够；如果设备不是 Jetson，不能使用 JetPack、
> `nvpmodel`、`jetson_clocks` 或本手册中的 DLA 配置。

## 1. 与桌面 x86_64 配置的关键区别

| 项目 | 桌面开发机 | G1 板载真机 |
| --- | --- | --- |
| CPU 架构 | `x86_64` | `aarch64` |
| 系统 | Ubuntu 22.04 / WSL2 | 原生 Ubuntu 22.04（L4T） |
| NVIDIA 软件栈 | 桌面驱动 + CUDA Toolkit | JetPack 6 提供驱动与 CUDA |
| CUDA | 通常 12.4 | JetPack 6.2 通常为 12.6 |
| TensorRT | **10.13，x86_64** | **10.7，aarch64** |
| ONNX Runtime 包 | `linux-x64` | `linux-aarch64` |
| Unitree DDS 库目录 | `x86_64` | `aarch64` |
| 启动方式 | `deploy.sh sim` | `deploy.sh real` 或指定真机网卡 |
| MuJoCo / 图形界面 | 需要 | 不需要 |

**不要把桌面端的 TensorRT 10.13、x86_64 ONNX Runtime、CUDA 软件源或已编译
二进制复制到 Orin 上使用。** C++ 程序和 TensorRT `.trt` 缓存必须在 Orin 上
重新生成。

## 2. 推荐并要求的版本

| 组件 | 版本/要求 |
| --- | --- |
| 设备 | Unitree G1 板载 NVIDIA Jetson Orin NX |
| CPU 架构 | `aarch64` |
| 操作系统 | Ubuntu 22.04 |
| JetPack | JetPack 6；仓库刷机文档当前使用 6.2 |
| CUDA | JetPack 自带 CUDA 12.6；不要另外安装桌面驱动 |
| TensorRT | **10.7 ARM64，必须使用该版本** |
| Python | Ubuntu 22.04 自带 Python 3.10，用于下载权重 |
| ONNX Runtime | 1.16.3 aarch64，由 `install_deps.sh` 安装 |
| Git | 支持 Git LFS |

官方部署说明明确要求 G1 板载 Orin 使用 TensorRT 10.7。不要使用系统里其他
版本的 TensorRT，也不要用桌面端要求的 10.13。TensorRT 版本不匹配可能造成
planner 输出错误，在真机上属于安全风险。

## 3. 上机前的安全要求

首次真机运行前必须做到：

1. 已经完整跑通桌面端 MuJoCo Sim2Sim；
2. 机器人使用吊架或其他可靠防跌落措施，双脚可以离地；
3. 机器人周围至少留出 3 米安全区；
4. 一名操作员只负责观察机器人并随时急停；
5. 部署终端保持在前台，确认按 `O` 可以停止控制；
6. 第一次只验证站立和平缓动作，不直接测试跑、跳、蹲起或遥操作；
7. 不同时运行 MuJoCo 和真机部署实例，避免 DDS 冲突。

`O` 是 SONIC 部署程序的紧急停止键，但不能代替机器人硬件急停、吊架和现场
安全员。

## 4. 检查系统底座和 CUDA Toolkit 是否已安装

SSH 登录 G1 板载 Orin 后执行：

```bash
uname -m
lsb_release -a
cat /etc/nv_tegra_release
```

预期：

```text
uname -m  -> aarch64
系统       -> Ubuntu 22.04
L4T       -> R36.x（JetPack 6）
```

分别检查 L4T 核心包和 JetPack 元包。不要隐藏错误信息：

```bash
dpkg-query -W nvidia-l4t-core
```

```bash
dpkg-query -W nvidia-jetpack
```

`nvidia-l4t-core` 证明 Jetson Linux/L4T 系统底座存在；`nvidia-jetpack` 是用于
安装 CUDA、TensorRT、cuDNN 等计算组件的元包。某些宇树定制镜像只有前者，
因此“系统是 JetPack 6 对应的 Ubuntu 22.04”并不自动等于“CUDA Toolkit 已经
完整安装”。

检查 CUDA Toolkit 的实际文件，而不是先检查环境变量：

```bash
ls -ld /usr/local/cuda*
command -v nvcc || true
test -x /usr/local/cuda/bin/nvcc \
  && /usr/local/cuda/bin/nvcc --version
test -x /usr/local/cuda-12.6/bin/nvcc \
  && /usr/local/cuda-12.6/bin/nvcc --version
```

再检查已安装的 CUDA/TensorRT 包和运行库：

```bash
dpkg -l | grep -E 'cuda-(toolkit|compiler|cudart)|nvinfer|cudla' || true
```

```bash
ldconfig -p | grep -E 'libcuda|libcudart|libnvinfer|libcudla' || true
```

检查 Jetson 工具：

```bash
command -v tegrastats
command -v jetson_clocks
```

可在另一个终端观察温度、频率和显存：

```bash
tegrastats
```

Jetson 上以 `tegrastats`、L4T 和 JetPack 信息为准，不把桌面端的
`nvidia-smi` 作为必要验收条件。

如果 `/etc/nv_tegra_release` 不存在，或系统仍是 JetPack 5 / L4T R35，先按
仓库官方 `docs/source/references/jetpack6.md` 完成 JetPack 6 升级，再继续。
刷写系统会改写存储设备，本手册不重复给出刷机命令。

### 4.1 如何判断下一步

按以下判断，不要跳步：

| 检查结果 | 含义 | 下一步 |
| --- | --- | --- |
| L4T 为 R36.x，`nvcc` 为 12.6，`/usr/local/cuda-12.6` 存在 | JetPack 6 和 CUDA Toolkit 已具备 | 跳到第 6 节 |
| L4T 为 R36.x，但 `nvcc` 不存在，且没有 `/usr/local/cuda*` | 只有 Jetson Linux 底座，缺少计算组件 | 执行第 5 节 |
| 有 `/usr/local/cuda-12.6`，但 `nvcc` 不在 PATH | CUDA 已装，环境变量尚未配置 | 跳到第 7 节 |
| L4T 为 R35.x | 仍是 JetPack 5 系统 | 先按官方指南升级/刷入 JetPack 6 |
| 不是 Jetson 或不是 aarch64 | 不适用本手册 | 停止安装并重新确认设备 |

以当前实际截图为例：

```text
aarch64                         -> 正确
Ubuntu 22.04                    -> 正确
L4T R36.4.3                     -> JetPack 6.2 系统底座正确
nvidia-l4t-core 36.4.3          -> L4T 核心包已安装
nvcc: command not found         -> CUDA 编译器当前不可用
/usr/local/cuda: No such file   -> 至少没有 CUDA 通用路径
```

`lsb_release` 显示的 `No LSB modules are available.` 只是说没有安装
可选的 LSB 兼容模块，不表示 Ubuntu、JetPack 或 CUDA 损坏，可以忽略。

因此当前还没有到“配置 CUDA 环境变量”的步骤。应先继续检查
`/usr/local/cuda-12.6`；如果版本目录也不存在，则按第 5 节补装 JetPack 计算
组件。仅修改 `PATH` 或 `.bashrc` 不能解决实际文件缺失。

## 5. 缺少 CUDA 时补齐 JetPack 6.2 计算组件

只有第 4 节判断为 CUDA Toolkit 确实缺失时才执行本节。L4T 已经是 R36.4.3
时不需要重新刷机。

### 5.1 安装前检查磁盘和 Jetson 软件源

```bash
df -h /
```

查看 NVIDIA Jetson 软件源：

```bash
cat /etc/apt/sources.list.d/nvidia-l4t-apt-source.list
```

L4T 36.4 / Orin 的源应指向 NVIDIA Jetson 的 `r36.4`，并包含 `common` 和
`t234`。不要添加桌面 `x86_64` CUDA 源。

刷新软件包索引：

```bash
sudo apt update
```

检查候选版本：

```bash
apt-cache policy nvidia-jetpack
```

```bash
apt-cache policy cuda-toolkit-12-6
```

必须能看到实际 `Candidate`。如果显示 `Candidate: (none)`，先修复 Jetson
`r36.4/t234` 软件源，不要改用桌面 CUDA 安装包。

### 5.2 推荐方案：安装完整 JetPack 计算组件

对于当前 G1 Orin，推荐使用：

```bash
sudo apt install -y nvidia-jetpack
```

这会按照当前 L4T 36.4 软件源补齐匹配的 CUDA、TensorRT、cuDNN 和 Jetson
开发组件，不会把系统变成 x86 桌面驱动环境。它可能占用较多磁盘空间，安装前
必须先检查根分区。

> JetPack 6.2 默认安装的系统 TensorRT 通常是 10.3。它用于补齐系统组件；本
> 项目要求的 TensorRT 10.7 仍在第 8 节使用独立 TAR 安装，两者不要混淆。

### 5.3 磁盘不足时的备选方案

只有确认无法安装完整 `nvidia-jetpack` 时，才先安装固定版本 CUDA Toolkit：

```bash
sudo apt install -y cuda-toolkit-12-6



# 后续如果需要再单独安装
sudo apt install -y nvidia-l4t-dla-compiler libcudla-dev-12-6
```

该元包安装 CUDA 12.6 开发工具，不安装桌面 NVIDIA 驱动。但后续仍需按项目
需要逐项补充 DLA、TensorRT 和其他依赖；当前仓库的 `install_deps.sh` 在
Jetson 上也可能继续建议安装完整 `nvidia-jetpack`，所以优先使用第 5.2 节。

### 5.4 安装后立即验证，不要先写环境变量

```bash
dpkg-query -W nvidia-jetpack || true
dpkg-query -W cuda-toolkit-12-6 || true
ls -ld /usr/local/cuda*
```

```bash
/usr/local/cuda-12.6/bin/nvcc --version
```

```bash
test -f /usr/local/cuda-12.6/include/cuda_runtime.h \
  && echo "CUDA 12.6 headers OK"
```

```bash
find /usr/local/cuda-12.6 -name 'libcudart.so*' -print
```

只有 `nvcc`、头文件和 `libcudart` 都存在，才进入第 6、7 节。

## 7. 验证并配置 CUDA 12.6 环境变量

本节只负责“配置已经安装好的 CUDA”，不负责下载 CUDA。先检查实际路径：

```bash
ls -ld /usr/local/cuda*
/usr/local/cuda-12.6/bin/nvcc --version
```

如果 `/usr/local/cuda` 已指向 `/usr/local/cuda-12.6`，为当前终端设置：

```bash
export CUDA_HOME=/usr/local/cuda
export CUDAToolkit_ROOT=/usr/local/cuda
export PATH="$CUDA_HOME/bin:$PATH"
export LD_LIBRARY_PATH="$CUDA_HOME/lib64:$CUDA_HOME/targets/aarch64-linux/lib:${LD_LIBRARY_PATH:-}"
```

如果只有 `/usr/local/cuda-12.6`，没有 `/usr/local/cuda` 通用软链接，则不要
猜测路径，直接使用版本目录：

```bash
export CUDA_HOME=/usr/local/cuda-12.6
export CUDAToolkit_ROOT=/usr/local/cuda-12.6
export PATH="$CUDA_HOME/bin:$PATH"
export LD_LIBRARY_PATH="$CUDA_HOME/lib64:$CUDA_HOME/targets/aarch64-linux/lib:${LD_LIBRARY_PATH:-}"
```

检查：

```bash
nvcc --version
test -f "$CUDA_HOME/include/cuda_runtime.h" && echo "CUDA headers OK"
find "$CUDA_HOME" -name 'libcudart.so*' -print
```

确认上述命令全部成功后，才把与实际路径一致的四行 `export` 加入当前用户：

```bash
nano ~/.bashrc
```

保存后加载并复查：

```bash
source ~/.bashrc
echo "$CUDA_HOME"
nvcc --version
```

## 8. 安装 TensorRT 10.7 ARM64

从 NVIDIA Developer 下载 **TensorRT 10.7、Linux aarch64/JetPack、CUDA 12.6
对应的 TAR 包**。不要下载 `x86_64`、`Windows`、`10.13` 或仅包含 Python
wheel 的包。

下载文件名通常类似：

```text
TensorRT-10.7.x.x.Linux.aarch64-gnu.cuda-12.6.tar.gz
```

如果在开发电脑上下载，可复制到 Orin：

```bash
scp ~/Downloads/TensorRT-10.7*.Linux.aarch64-gnu.cuda-12.6.tar.gz \
  <G1用户名>@<G1-IP>:~/
```

以下命令在 **Orin** 上执行。先确认包名确实包含 `aarch64`：

```bash
uname -m
ls -lh ~/TensorRT-10.7*.Linux.aarch64-gnu.cuda-12.6.tar.gz
```

解压并建立稳定路径：

```bash
mkdir -p ~/opt
pv ~/TensorRT-10.7*.Linux.aarch64-gnu.cuda-12.6.tar.gz | tar -xz -C ~/opt
```

查看实际目录名：

```bash
find ~/opt -maxdepth 1 -type d -name 'TensorRT-10.7*' -print
```

将下方右侧目录替换为实际解压出的目录：

```bash
ln -sfn ~/opt/TensorRT-10.7.x.x ~/TensorRT
```

设置当前终端：

```bash
export TensorRT_ROOT="$HOME/TensorRT"
export PATH="$TensorRT_ROOT/bin:$CUDA_HOME/bin:$PATH"
export LD_LIBRARY_PATH="$TensorRT_ROOT/lib:$CUDA_HOME/lib64:$CUDA_HOME/targets/aarch64-linux/lib:/usr/lib/aarch64-linux-gnu/nvidia:${LD_LIBRARY_PATH:-}"
```

确认头文件、库、架构和版本：

```bash
test -f "$TensorRT_ROOT/include/NvInfer.h" && echo "NvInfer.h OK"
find "$TensorRT_ROOT/lib" -maxdepth 1 -name 'libnvinfer.so*' -print
file "$TensorRT_ROOT/lib/libnvinfer.so"*
grep -E 'NV_TENSORRT_(MAJOR|MINOR|PATCH|BUILD)' \
  "$TensorRT_ROOT/include/NvInferVersion.h"
```

必须确认：

- 版本是 `10.7.x.x`；
- `file` 输出包含 `ARM aarch64`，而不是 `x86-64`；
- `NvInfer.h`、`libnvinfer.so` 和 `libnvonnxparser.so` 存在。

验证 `trtexec` 可以加载动态库：

```bash
"$TensorRT_ROOT/bin/trtexec" --version
ldd "$TensorRT_ROOT/bin/trtexec" | grep 'not found' || true
```

若 `ldd` 没有输出 `not found`，动态库解析正常。

确认无误后，将以下内容加入 Orin 用户的 `~/.bashrc`：

```bash
export TensorRT_ROOT="$HOME/TensorRT"
export PATH="$TensorRT_ROOT/bin:$CUDA_HOME/bin:$PATH"
export LD_LIBRARY_PATH="$TensorRT_ROOT/lib:$CUDA_HOME/lib64:$CUDA_HOME/targets/aarch64-linux/lib:/usr/lib/aarch64-linux-gnu/nvidia:${LD_LIBRARY_PATH:-}"
```

这里只追加 TensorRT 配置，不要重写第 7 节已根据真实目录设置的
`CUDA_HOME` 和 `CUDAToolkit_ROOT`。如果真机只有
`/usr/local/cuda-12.6`，则继续保持这个版本目录，不要改回不存在的
`/usr/local/cuda`。

重新加载：

```bash
source ~/.bashrc
```

## 9. 在 Orin 上获取仓库

推荐在 Orin 上直接克隆，避免把开发机的 `build/`、`target/` 和 x86_64
TensorRT 缓存带到真机：

```bash
mkdir -p ~/g1_sonic
cd ~/g1_sonic
git clone https://github.com/NVlabs/GR00T-WholeBodyControl.git
cd GR00T-WholeBodyControl
git lfs pull
```

检查：

```bash
git lfs status

```

如果需要与开发电脑严格保持同一版本，在两台机器上分别运行
`git rev-parse HEAD`，确保提交号一致。

如果必须复制已有仓库，只复制源码、配置、ONNX 和参考动作；不要复制这些生成
物：

```text
gear_sonic_deploy/build/
gear_sonic_deploy/target/
任何 *.trt TensorRT engine 缓存
x86_64 的 /opt/onnxruntime
```

## 10. 下载 SONIC 部署权重

真机部署不需要 `.venv_sim` 和 MuJoCo。单独创建轻量下载环境：

```bash
cd ~/g1_sonic/GR00T-WholeBodyControl
python3 -m venv .venv_download
source .venv_download/bin/activate
python -m pip install --upgrade pip
python -m pip install huggingface_hub
python download_from_hf.py
deactivate
```

检查默认模型：

```bash
ls -lh gear_sonic_deploy/policy/release/model_encoder.onnx
ls -lh gear_sonic_deploy/policy/release/model_decoder.onnx
ls -lh gear_sonic_deploy/policy/release/observation_config.yaml
ls -lh gear_sonic_deploy/planner/target_vel/V2/planner_sonic.onnx
```

如需低延迟版本，额外执行：

```bash
source .venv_download/bin/activate
python download_from_hf.py --low-latency
deactivate
```

不要运行 `--training`；它会下载训练 checkpoint 和大体量 SMPL 数据，真机部署
不需要。

## 11. 安装 ARM64 C++ 依赖

进入部署目录并确认架构与环境变量：

```bash
cd ~/g1_sonic/GR00T-WholeBodyControl/gear_sonic_deploy
uname -m
echo "$CUDA_HOME"
echo "$TensorRT_ROOT"
nvcc --version
```

预期 `uname -m` 为 `aarch64`，TensorRT 为 10.7。然后运行仓库脚本：

```bash
chmod +x scripts/install_deps.sh
./scripts/install_deps.sh
```

该脚本会自动识别 Jetson/aarch64，并完成：

- 安装 C++ 编译依赖；
- 下载 `onnxruntime-linux-aarch64-1.16.3` 到 `/opt/onnxruntime`；
- 下载 aarch64 版本的 `just`；
- 使用仓库自带的 Unitree SDK2 aarch64 库；
- 检查 CUDA，并在 Jetson 上补充 JetPack 开发组件。

不要把开发机 `/opt/onnxruntime` 复制到 Orin。脚本完成后检查：

```bash
file /opt/onnxruntime/lib/libonnxruntime.so*
file thirdparty/unitree_sdk2/lib/aarch64/libunitree_sdk2.a
file thirdparty/unitree_sdk2/thirdparty/lib/aarch64/libddsc.so
just --version
```

输出应为 ARM/aarch64。

## 12. 加载环境并在 Orin 上编译

确保没有激活 Conda 或 Python venv：

```bash
conda deactivate 2>/dev/null || true
deactivate 2>/dev/null || true
```

加载仓库环境：

```bash
cd ~/g1_sonic/GR00T-WholeBodyControl/gear_sonic_deploy
source scripts/setup_env.sh
```

该脚本会优先加入仓库内匹配的 Unitree CycloneDDS 库：

```text
gear_sonic_deploy/thirdparty/unitree_sdk2/thirdparty/lib/aarch64
```

检查实际解析的关键库：

```bash
echo "$onnxruntime_DIR"
echo "$CUDAToolkit_ROOT"
echo "$TensorRT_ROOT"
echo "$LD_LIBRARY_PATH" | tr ':' '\n'
```

编译：

```bash
just build
```

检查生成物：

```bash
file target/release/g1_deploy_onnx_ref
ldd target/release/g1_deploy_onnx_ref | grep 'not found' || true
```

`file` 必须显示 aarch64；`ldd` 不应显示缺失库。

> 如果仓库是从 x86_64 电脑整体复制来的，先确认 `build/` 和 `target/` 中没有
> 桌面端产物，再在 Orin 上执行 `just clean && just build`。`just clean` 会删除
> `gear_sonic_deploy/build/` 和 `gear_sonic_deploy/target/`，不会删除源码和权重。

## 13. 配置 G1 真机网络

SONIC 通过 Unitree SDK2 / DDS 与机器人通信。先查看接口：

```bash
ip -br link
ip -br -4 addr
```

`deploy.sh real` 会优先查找具有 `192.168.123.x` 地址的接口。记录实际接口名，
例如 `eth0` 或 `enp5s0`：

```bash
ip -o -4 addr show | grep '192\.168\.123\.'
```

如果板载 Orin 的机器人侧接口已经有 `192.168.123.x/24` 地址，不要重复修改。
如果没有地址，需按 Unitree G1 SDK 网络说明设置与机器人同网段的静态地址，并
确保不与已有设备冲突。示例中的接口和 IP 必须替换为现场实际值：

```bash
sudo ip addr add 192.168.123.222/24 dev <机器人侧网卡>
sudo ip link set <机器人侧网卡> up
```

再次检查：

```bash
ip -br -4 addr show <机器人侧网卡>
```

不要把 Wi-Fi、互联网接口或回环接口 `lo` 误传给真机部署。第一次运行建议显式
写接口名，不依赖自动选择。

## 14. 真机部署前的只读检查

此阶段机器人保持吊起，不启动控制策略。

检查是否有可能冲突的部署进程：

```bash
pgrep -af 'run_sim_loop|g1_deploy_onnx_ref|deploy.sh' || true
```

检查 DDS/ROS 环境残留：

```bash
env | grep -E 'CYCLONEDDS|ROS_DOMAIN|RMW_IMPLEMENTATION|FASTRTPS' || true
```

检查默认部署文件来自同一 release：

```bash
cd ~/g1_sonic/GR00T-WholeBodyControl/gear_sonic_deploy
test -f policy/release/model_encoder.onnx
test -f policy/release/model_decoder.onnx
test -f policy/release/observation_config.yaml
test -f planner/target_vel/V2/planner_sonic.onnx
```

最后检查动态库：

```bash
source scripts/setup_env.sh
ldd target/release/g1_deploy_onnx_ref | grep 'not found' || true
```

只有以上检查全部通过，才进入真机启动阶段。

## 15. 首次启动 SONIC 真机控制

先让安全员到位、清空 3 米范围并把机器人可靠吊起。部署终端必须保持在前台。

推荐首次显式指定机器人侧网卡，并固定使用键盘输入，避免外部 ZMQ、手柄或 ROS 2
命令介入首次验收：

```bash
cd ~/g1_sonic/GR00T-WholeBodyControl/gear_sonic_deploy
source scripts/setup_env.sh
./deploy.sh --input-type keyboard <机器人侧网卡>
```

确认启动器打印的内容：

```text
Environment:        real
Network Interface:  <机器人侧网卡>
Decoder Model:      policy/release/model_decoder.onnx
Encoder Model:      policy/release/model_encoder.onnx
Obs Config:         policy/release/observation_config.yaml
Planner:            planner/target_vel/V2/planner_sonic.onnx
Input Type:         keyboard
```

确认无误后才在 `Proceed with deployment? [Y/n]:` 输入 `Y`。也可以使用自动检测：

```bash
./deploy.sh --input-type keyboard real
```

但如果日志出现：

```text
Could not find 192.168.123.x interface, using: ...
```

应取消启动，查明并显式传入正确网卡，不要接受未知的 fallback 接口。

首次启动时会在 Orin 上把 ONNX 转换为 TensorRT `.trt` engine，耗时会比后续
启动长。不要使用从 x86_64、其他 GPU 或其他 TensorRT 版本生成的 `.trt` 文件。

### 首次动作顺序

1. 机器人保持吊起，观察终端直到初始化完成；
2. 不要立即播放动态参考动作；
3. 按仓库当前键盘说明使用 `]` 启动策略；
4. 先验证关节方向、站姿、抖动、温度和 DDS 频率；
5. 任一关节方向异常、持续振荡、状态延迟或温度异常，立即按 `O`；
6. 吊起状态验证通过后，再在现场安全流程下逐步进行承重测试；
7. 低速站立稳定后，才测试平缓参考动作。

紧急停止：

```text
键盘 O：立即停止控制并退出
```

同时始终准备使用 G1 的硬件急停方案。

## 18. 常见问题

### 18.1 `Exec format error`

通常加载了 x86_64 程序或库：

```bash
file target/release/g1_deploy_onnx_ref
file /opt/onnxruntime/lib/libonnxruntime.so*
file "$TensorRT_ROOT/lib/libnvinfer.so"*
```

所有输出都应为 ARM/aarch64。清理复制来的 x86 构建产物后，在 Orin 上重新运行
`just build`。

### 18.2 TensorRT 版本不是 10.7

检查实际头文件与运行库：

```bash
grep -E 'NV_TENSORRT_(MAJOR|MINOR|PATCH|BUILD)' \
  "$TensorRT_ROOT/include/NvInferVersion.h"
ldd target/release/g1_deploy_onnx_ref | grep nvinfer
```

如果链接到了 `/usr/lib/aarch64-linux-gnu` 中的其他 TensorRT 版本，确认
`$TensorRT_ROOT/lib` 位于 `LD_LIBRARY_PATH` 前部，重新 `source scripts/setup_env.sh`
并重新编译。

### 18.3 `libcudla.so.1`、DLA 或 CUDA 库找不到

```bash
sudo apt update
sudo apt install -y nvidia-l4t-dla-compiler libcudla-dev-12-6
source scripts/setup_env.sh
ldconfig -p | grep -E 'cudla|nvinfer|cudart'
```

`setup_env.sh` 会为 Jetson 添加 `/usr/lib/aarch64-linux-gnu/nvidia`，并在需要时
建立兼容软链接。

### 18.4 `nvcc` 或 `cuda_runtime.h` 找不到

```bash
ls -l /usr/local/cuda*
find /usr/local/cuda-12.6 -name cuda_runtime.h -print 2>/dev/null
dpkg-query -W nvidia-jetpack || true
dpkg-query -W cuda-toolkit-12-6 || true
apt-cache policy nvidia-jetpack cuda-toolkit-12-6
```

如果 `/usr/local/cuda-12.6` 也不存在，说明不是单纯的 `PATH`问题，
返回第 5 节安装 JetPack 计算组件。不要添加 x86_64 CUDA 软件源。

### 18.5 `ChannelFactory create domain error` 或收不到机器人状态

检查网卡和地址：

```bash
ip -br -4 addr
ip -o -4 addr show | grep '192\.168\.123\.'
```

显式运行：

```bash
./deploy.sh --input-type keyboard <机器人侧网卡>
```

同时确认没有第二个仿真/部署实例，并检查 DDS/ROS 环境变量是否污染当前终端。

### 18.6 CycloneDDS 崩溃或堆损坏

ROS 2 自带的 CycloneDDS 可能与 Unitree SDK2 捆绑版本冲突。重新加载：

```bash
cd ~/g1_sonic/GR00T-WholeBodyControl/gear_sonic_deploy
source scripts/setup_env.sh
echo "$LD_LIBRARY_PATH" | tr ':' '\n' | head
```

确保仓库的以下目录排在 ROS 2 DDS 库之前：

```text
thirdparty/unitree_sdk2/thirdparty/lib/aarch64
```

### 18.7 首次启动很慢

首次运行会从 ONNX 构建 Orin 专用 TensorRT `.trt` engine，属于正常现象。若
TensorRT、CUDA、GPU 或精度参数变化，应重新生成缓存，不能继续使用旧 engine。

### 18.8 编译过程中内存不足或温度过高

观察：

```bash
tegrastats
```

`just build` 默认使用 `nproc` 的全部核心并行编译。如果出现 OOM，可暂时手动
降低并行度：

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j2
```

温度过高时停止编译并检查风扇、散热和性能模式，不要在降频或过热状态下进行
实时控制测试。

### 18.9 模型加载成功但动作异常

立即按 `O`，并依次核对：

1. TensorRT 是否严格为 ARM64 10.7；
2. encoder、decoder、observation config 是否为同一模型版本；
3. planner 是否为同一次官方下载；
4. 是否误用了从其他机器复制的 `.trt` engine；
5. 真机型号、DOF 和观测配置是否匹配；
6. DDS 状态频率、关节映射和起始姿态是否正常。

未定位原因前不要再次让机器人落地承重。
