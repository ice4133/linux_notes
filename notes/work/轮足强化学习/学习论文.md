## 一、先建立完整的知识地图

当前最重要的事情，不是继续用“遇到一个问题就问一个问题”的方式零散学习，而是先建立轮足机器人从模型到真机的完整链路：

```text
机器人模型
   ↓
单关节控制
   ↓
单腿运动学
   ↓
整机状态估计
   ↓
站立控制
   ↓
轮式运动
   ↓
腿轮协同
   ↓
强化学习训练
   ↓
仿真验证
   ↓
真机测试
```

GitHub 上很难找到一个项目从零覆盖全部环节。更现实的方法是选择几个各有所长的仓库，把它们拼成一套系统课程：

- 用资源索引建立领域地图；
- 用传统四足框架理解软件分层；
- 用成熟机器人项目理解有限状态机与硬件抽象；
- 用轮式四足 RL 项目理解训练、回放、Sim2Sim 和 Sim2Real；
- 用论文理解轮子加入四足后带来的滚动约束、状态估计与全身控制问题；
- 最后在自己的 MuJoCo 工程中逐级实现和验证。

## 二、推荐的 GitHub 学习资源

### 2.1 总索引：Awesome Wheel-Legged Robots

仓库：[XinLang2019/awesome-wheeled-legged](https://github.com/XinLang2019/awesome-wheeled-legged)

这是最适合作为入口收藏的轮腿机器人资源目录。内容包括：

- 开源项目；
- 模型控制论文；
- 强化学习论文；
- 硬件平台；
- 轮腿机器人综述资料。

它适合作为“轮足机器人知识目录”，但不是可以直接运行的完整工程。第一遍不需要读完全部内容，只需要浏览项目分类、论文标题和硬件平台，知道轮足领域有哪些主要模块。

**建议输出：**自己整理一张领域地图，把感兴趣的项目按“模型控制、强化学习、硬件、状态估计、规划”分类。

---

### 2.2 学整机软件结构：CHAMP

仓库：[chvmp/champ](https://github.com/chvmp/champ)

CHAMP 主要面向普通四足，而不是四轮足，但非常适合学习以下内容：

- URDF 怎样接入控制器；
- 步态发生器是什么；
- 足端轨迹怎样生成；
- 逆运动学放在哪一层；
- 仿真和真机怎样共用控制逻辑；
- 硬件接口怎样与上层控制解耦；
- ROS 节点之间怎样组织。

阅读时重点不是让它先跑起来，而是理解它的软件分层：

```text
速度命令
   ↓
步态规划
   ↓
足端轨迹
   ↓
逆运动学
   ↓
关节目标
   ↓
硬件接口
   ↓
电机
```

四轮足机器人还需要增加一条轮式控制链路：

```text
机体速度命令
   ↓
轮速分配
   ↓
轮电机目标速度或目标力矩
```

CHAMP 还提供机器人配置辅助工具，可以帮助理解自研机器人如何接入一个整机框架。

**建议输出：**画出 CHAMP 的信息流图，并标出自己的轮足机器人需要新增或替换的模块。

### 2.3 学成熟控制软件架构：MIT Cheetah Software

仓库：[mit-biomimetics/Cheetah-Software](https://github.com/mit-biomimetics/Cheetah-Software)  
类与文件文档：[Cheetah Software Documentation](https://mit-biomimetics.github.io/index.html)

这个项目不适合作为第一个运行项目，但非常适合阅读成熟机器人软件的架构。建议重点查看：

```text
RobotRunner
HardwareBridge
LegController
StateEstimator
ControlFSM
FSM_State_Passive
FSM_State_StandUp
FSM_State_BalanceStand
FSM_State_Locomotion
```

它最有价值的思想是：成熟机器人不会把全部控制代码堆在一个循环里，而是使用有限状态机组织整机行为。

```text
PASSIVE
   ↓
JOINT_TEST
   ↓
STAND_UP
   ↓
BALANCE_STAND
   ↓
LOCOMOTION
   ↓
RECOVERY
```

自研四轮足也应建立类似状态：

```cpp
enum class RobotMode
{
    PASSIVE,
    MOTOR_TEST,
    LEG_TEST,
    WHEEL_TEST,
    STAND_UP,
    STAND,
    DRIVE,
    LEG_WHEEL_LOCOMOTION,
    RL_CONTROL,
    EMERGENCY_STOP
};
```

**建议输出：**在自己的工程里先建立有限状态机骨架，即使每个状态暂时只输出日志，也要先把进入、运行、退出和故障转移逻辑设计清楚。

### 2.4 最接近目标的项目：Go2W RL Gym

仓库：[ShengqianChen/go2w_rl_gym](https://github.com/ShengqianChen/go2w_rl_gym)

这是很值得仔细阅读的轮式四足强化学习仓库。它覆盖了：

- Go2W 仿真环境；
- RL 训练；
- 策略回放；
- MuJoCo Sim2Sim；
- 实物部署；
- 站立测试示例；
- `unitree_sdk2_python` 通信接口。

项目体现了下面这条完整流程：

```text
Train
  ↓
Play
  ↓
Sim2Sim
  ↓
Sim2Real
```

也就是先训练，再在原训练仿真器中回放，然后部署到 MuJoCo 做跨仿真器验证，最后才进入真机。

仓库中还提供站立测试示例，例如：

```text
test/go2w_stand_example.py
```

这说明完整工程不会一上来就运行 RL，而是先完成确定性的站立验证。

阅读时重点关注：

```text
机器人关节名称和顺序
默认关节角
observation 的组成
action 如何转成关节控制命令
reward 的组成
deploy 中的控制循环
```

不要直接把 Go2W 的参数复制给自研机器人。下面这些都可能不同：

- 关节数量与顺序；
- 关节方向；
- 质量与惯量；
- 默认姿态；
- 腿部结构；
- 电机限制；
- 轮子半径。

应该复制的是它的**软件流程与验证方法**，而不是硬件参数。

### 2.5 新一些的轮足 RL 项目：DDT_Lab

仓库：[DDTRobot/DDT_Lab](https://github.com/DDTRobot/DDT_Lab)

该项目基于 Isaac Lab，包含 DDT D1 与 Tita 等轮腿机器人，可用于后期研究：

- 轮足 RL 环境如何组织；
- 约束强化学习；
- 轮腿机器人的奖励设计；
- Isaac Lab 项目结构；
- 自定义 URDF 如何接入训练环境。

现阶段不建议把它作为唯一主线，因为 Isaac Sim、Isaac Lab、CUDA 等环境本身就有较大的配置和学习成本。如果当前已经打通 MuJoCo 数据链路，应先沿用 MuJoCo 完成基础控制和测试，不要为了学习资料频繁更换仿真器。

### 2.6 学仿真与控制接口分离：Unitree MuJoCo

仓库：[unitreerobotics/unitree_mujoco](https://github.com/unitreerobotics/unitree_mujoco)

这个项目最值得学习的不是宇树机器人的具体模型，而是控制接口与仿真器分离的思想：

```text
控制程序
   ↕ LowCmd / LowState
仿真接口
   ↕
MuJoCo
```

控制器只依赖统一的机器人状态和控制命令，不应该关心数据究竟来自 MuJoCo、ROS、DDS、CAN 还是 RS485。

自研机器人可以设计类似接口：

```cpp
struct RobotState
{
    JointState joints[NUM_JOINTS];
    ImuState imu;
    BaseState base;
};

struct RobotCommand
{
    JointCommand joints[NUM_JOINTS];
};
```

上层控制器只接受：

```cpp
RobotState state;
```

并输出：

```cpp
RobotCommand command;
```

这就是“像整机一样开发”的核心：同一个控制器可以运行在仿真和实物上，底层适配层负责处理数据来源与通信方式的差异。

## 三、论文阅读顺序

论文不要按照搜索结果随机阅读。建议按照“轮式约束 → 混合运动 → 状态估计 → 强化学习”的顺序建立概念。

### 3.1 第一篇：Keep Rollin’

**Keep Rollin’ – Whole-Body Motion Control and Planning for Wheeled Quadrupedal Robots**

重点理解：

- 轮子不是普通足端；
- 轮子存在非完整滚动约束；
- 轮子速度与机体速度如何联系；
- 腿部姿态和轮子滚动如何统一到全身控制；
- 为什么不能只在普通四足控制器后面简单增加轮电机速度。

论文使用轨迹优化和分层全身控制，并考虑轮子的无侧滑滚动约束。第一次阅读不要求推导全部优化公式，先看：

1. 系统框图；
2. 控制器输入与输出；
3. 轮地约束如何进入模型；
4. 实验任务和评价方式。

---

### 3.2 第二篇：Rolling in the Deep

**Rolling in the Deep – Hybrid Locomotion for Wheeled-Legged Robots Using Online Trajectory Optimization**

重点理解轮腿机器人的三类运动模式：

```text
Driving：轮子始终接触地面，主要依靠滚动
Stepping：轮子离开地面，腿部执行迈步
Hybrid：行驶与迈步混合
```

这篇论文帮助建立一个关键认识：**轮足机器人不等于“带轮子的普通四足”**。它需要同时规划轮子轨迹、机体轨迹和接触切换，并通过全身控制输出关节力矩。

当前阶段不必立刻复现论文，重点理解系统模块之间的关系，以及粗糙地形上为什么必须混合行驶与迈步。

---

### 3.3 第三篇：轮足状态估计

**State Estimation for Hybrid Locomotion of Driving-Stepping Quadrupeds**

普通四足常利用“接触足速度近似为零”的假设估计机体速度，但轮足机器人不同：

```text
轮子接触地面
≠
接触点对应的轮轴速度为零
```

轮子滚动时，轮速、轮半径、轮轴运动、接触几何和可能发生的滑移都需要进入状态估计模型。这篇论文与以下内容直接相关：

- IMU 融合；
- 轮速里程计；
- 机体位置和速度估计；
- 驱动与迈步模式切换；
- 卡尔曼滤波中的观测模型。

阅读时重点回答三个问题：

1. 状态向量中包含什么；
2. 轮速怎样变成观测量；
3. 轮子打滑或离地时，观测可信度怎样变化。

---

### 3.4 第四篇：轮足强化学习

**Control of Wheeled-Legged Quadrupeds Using Deep Reinforcement Learning**

这类工作用于理解 RL 如何同时控制腿关节和轮关节，而不是给普通四足 RL 简单增加四个输出。

建议在前三类论文之后再阅读，否则很容易看到下面这些概念，却不知道它们为什么如此设计：

```text
observation
action
reward
domain randomization
```

阅读时重点记录：

- observation 中哪些量来自 IMU、编码器和命令；
- action 是关节位置、速度还是力矩；
- 腿和轮的 action 是否采用不同控制方式；
- 奖励如何平衡速度跟踪、姿态稳定、能耗和滑移；
- 仿真参数如何随机化；
- 真机上有哪些安全限制。

## 四、整机测试分级流程

不要把“机器人能不能走”作为第一个测试目标。正确方式是从模型、单关节、单腿和单轮开始，逐步增加系统耦合度。

### 第 0 级：模型检查

只检查 MuJoCo 模型，不运行运动控制器。

需要确认：

- 每个关节的名称；
- 每个关节的正方向；
- 编码器零位和模型零位；
- 关节上下限；
- 轮子旋转轴；
- 质量与质心；
- 转动惯量；
- 碰撞体；
- 摩擦系数；
- 电机编号与仿真关节顺序。

建议输出一张完整表格：

| ID | 关节 | 正方向 | 零位 | 最小角 | 最大角 | 控制模式 |
|---:|---|---|---:|---:|---:|---|
| 0 | 左前腿关节 1 | 待实测 | 待标定 | 待填写 | 待填写 | MIT/位置 |
| 1 | 左前腿关节 2 | 待实测 | 待标定 | 待填写 | 待填写 | MIT/位置 |
| 2 | 左前轮 | 待实测 | 0 | 连续旋转 | 连续旋转 | 速度/力矩 |

**通过条件：**关节顺序、方向、零位、限位、轮轴、质量惯量和实物配置全部能够一一对应。在这里没有完全确认之前，不进入 RL。

---

### 第 1 级：单关节测试

机器人悬空，一次只控制一个关节。

依次测试：

```text
常值小力矩
位置阶跃
正弦位置
速度命令
关节限位
急停
通信超时
```

记录：

```text
timestamp
q_des
q
dq_des
dq
tau_cmd
tau_est
position_error
loop_dt
fault_code
```

合格标准不是“电机动了”，而是：

- 方向正确；
- 没有高频振荡；
- 没有持续超调；
- 没有越过限位；
- 控制周期稳定；
- 命令停止或通信超时后能够进入安全状态；
- 急停能够切断主动输出。

---

### 第 2 级：单腿测试

每次只测试一条腿。

测试内容：

1. 给定关节角，检查足端位置；
2. 给定足端位置，用逆运动学得到关节角；
3. 足端画直线；
4. 足端画圆；
5. 腿部上下伸缩；
6. 检查奇异位形和关节限位。

目标是验证：

```text
q → 正运动学 → 足端位置

足端目标 → 逆运动学 → q_des
```

如果腿部缺少侧向髋关节，足端只能在主要运动平面内活动。许多三自由度四足仓库中的三维足端控制器不能原样移植，但正运动学、逆运动学、轨迹跟踪和奇异位形测试方法仍然适用。

**通过条件：**正逆运动学互相验证，轨迹连续，无跳解，不越限，接近奇异位形时能够限速或拒绝命令。

---

### 第 3 级：单轮测试

腿保持固定，只测试一个轮子。

测试内容：

- 正转方向；
- 反转方向；
- 轮速阶跃；
- 低速稳定性；
- 空载速度；
- 接地速度；
- 轮地摩擦；
- 轮速与机体速度的关系。

然后逐步扩大为：

```text
单轮
  ↓
同侧两个轮子
  ↓
左右轮组
  ↓
四个轮子
```

**通过条件：**方向和编号完全正确；低速不爬行或振荡；左右轮速度一致；接地后没有异常电流、持续滑移或结构干涉。

---

### 第 4 级：固定姿态站立

这个阶段不要使用 RL。先使用确定性的关节 PD：

$$
\tau_i = K_{p,i}(q_{des,i}-q_i) + K_{d,i}(0-\dot q_i)
$$

让机器人从当前姿态平滑插值到默认站立姿态：

```text
current_q
   ↓ 2～5 秒平滑插值
default_stand_q
```

需要依次完成：

- 悬空站立姿态；
- 地面站立；
- 低刚度测试；
- 中等刚度测试；
- 外力轻推恢复；
- 重力补偿效果检查。

**通过条件：**插值过程中没有冲击；关节不饱和；机体姿态稳定；轻推后可以恢复；急停时机器人行为可预测。只有确定性站立稳定，才有资格训练 RL 站立和运动。

---

### 第 5 级：固定腿姿态轮式行驶

四条腿保持固定姿态，只使用轮子行驶。这个阶段可以把机器人视为一台四轮移动机器人。

先实现：

```text
前进
后退
停止
缓慢加速
缓慢减速
左右差速转向
```

测试：

- 速度跟踪误差；
- 左右轮速度是否一致；
- 直线是否跑偏；
- 急停距离；
- 转向振动；
- 轮胎打滑；
- IMU yaw 变化；
- 速度命令斜坡限制。

这个阶段不要引入腿部动态运动。

---

### 第 6 级：机体姿态控制

轮子停止，通过腿部关节控制：

```text
机体高度
roll
pitch
```

基本映射可以先从以下直觉建立：

```text
height command → 四腿共同伸缩
roll command   → 左右腿差动伸缩
pitch command  → 前后腿差动伸缩
```

先分别测试高度、roll、pitch，再测试组合命令。记录姿态响应、关节限位、机体高度误差和接触稳定性。

**通过条件：**单项和小范围组合命令均可稳定跟踪，无明显耦合振荡，不发生轮子离地或关节撞限位。完成这一层后，才开始边行驶边调整姿态。

---

### 第 7 级：轮腿协同

逐步测试：

```text
固定高度行驶
行驶中改变高度
行驶中调整 pitch
斜坡上保持机体水平
单个轮子跨越小障碍
前轮抬升
前后腿依次越障
```

这一阶段才是真正的四轮足控制。每次只增加一个变量，并保留上一级的安全限幅、日志和急停机制。

---

### 第 8 级：RL 环境

确定性控制全部验证后，再建立 RL 任务。

#### 最小 Observation

建议从最小集合开始：

```text
机体角速度
重力在机体坐标系中的投影
关节位置
关节速度
轮速
上一时刻 action
期望前进速度
期望 yaw 角速度
```

只有当现有观测无法解决明确问题时，再增加机体线速度、接触状态、地形高度等信息。

#### Action 设计

不要一开始就让策略直接输出所有电机力矩。更稳妥的起点是：

```text
腿关节目标位置修正量
+
轮子目标速度或轮子力矩
```

例如：

```text
q_des = q_default + action_leg * action_scale
wheel_cmd = action_wheel * wheel_scale
```

#### 环境核心函数

阅读和实现 RL 环境时，重点理解：

```python
step()
_compute_torques()
compute_observations()
compute_reward()
check_termination()
reset_idx()
```

这六类函数构成机器人 RL 环境的核心骨架：动作进入控制层，物理仿真前进，状态刷新，判断终止，计算奖励，再构造下一时刻观测。

---

### 第 9 级：压力测试

训练结果不能只用“视频里能跑”判断。必须建立统一测试集：

```text
不同速度命令
不同转向命令
不同摩擦系数
不同机体质量
不同负载
电机力矩下降
通信延迟
观测噪声
随机外力
斜坡
台阶
碎石地形
轮子打滑
某关节性能下降
```

记录统一指标：

```text
速度跟踪 RMSE
姿态角 RMSE
跌倒率
最大关节力矩
最大关节速度
能耗
轮子滑移率
扰动恢复时间
单次运行持续时间
安全限制触发次数
```

只有使用相同命令、随机种子、地形和评价指标，才能客观比较两个策略，而不是凭视频感觉判断。

## 五、自研机器人与开源轮足平台的关键区别

许多开源轮足平台默认每条腿具有较完整的三维运动能力。如果自研机构缺少侧向髋关节，需要特别注意：

- 足端不能任意做三维位置控制；
- 无法直接复制普通四足的落脚点规划；
- 横向平衡更依赖机体姿态和轮地作用；
- 转向可能主要依赖左右轮差速；
- 侧向抗扰能力可能弱于三自由度腿；
- RL 的动作空间和奖励必须按照实际机构重新设计。

因此，不能直接拿 Go2W 的策略控制自研机器人。

### 可以复用的部分

- RL 训练框架；
- PPO 等算法实现；
- observation 的组织方法；
- reward 的组织方法；
- domain randomization；
- 仿真与实物接口设计；
- 有限状态机；
- 日志系统；
- 分级测试流程；
- 策略导出方式；
- Sim2Sim 思想；
- 安全限幅机制。

### 必须按自研机器人重写或重新标定的部分

```text
URDF / MJCF
关节数量和顺序
关节正方向与零位
正运动学
逆运动学
雅可比
默认姿态
动作维度
轮子约束
奖励参数
电机限制
实物通信层
```

## 六、实际学习主线

不要同时运行六个仓库。建议按下面四个阶段推进。

### 第一阶段：建立知识地图

阅读：

1. `awesome-wheeled-legged` 的分类与论文列表；

4. *Keep Rollin’* 的系统框图和实验部分。

目标：知道一个整机控制系统包含哪些模块，并能画出自己的系统框图。

---

### 第二阶段：完成确定性测试程序

在当前 MuJoCo 工程中实现：

```text
PASSIVE
MOTOR_TEST
LEG_TEST
WHEEL_TEST
STAND
DRIVE
```

这些状态不使用 RL。目标是能够通过键盘、手柄或测试脚本安全切换状态，并稳定控制每个模块。

同时建立：

- 统一 `RobotState` / `RobotCommand`；
- 时间戳和控制周期统计；
- 数据日志；
- 关节位置、速度、力矩限幅；
- 命令超时保护；
- 急停；
- 故障码与状态转移规则。

---

### 第三阶段：阅读 RL 环境

重点阅读：

1. `legged_gym/envs/base/legged_robot.py`；
2. `go2w_rl_gym` 中的机器人配置；
3. observation；
4. reward；
5. action 到电机命令的转换；
6. MuJoCo 部署控制循环。

Go2W RL Gym 将训练、回放、MuJoCo Sim2Sim 和实物部署串成了一条完整流程，适合作为软件结构参考。

阅读时不要只逐行看代码。每看一个函数，都要回答：

```text
它属于哪一层？
输入是什么？
输出是什么？
以什么频率运行？
仿真和真机是否共用？
失败时怎样进入安全状态？
```

---

### 第四阶段：建立自研机器人的最小 RL 任务

不要一开始训练“复杂地形高速运动”。建议依次训练：

```text
任务 1：原地保持站立
任务 2：固定腿姿态轮式速度跟踪
任务 3：速度跟踪 + 姿态稳定
任务 4：行驶中改变机体高度
任务 5：轻微斜坡
任务 6：小障碍
任务 7：腿轮混合越障
```

每个任务稳定后，再增加下一个任务。每次改变 observation、action、reward 或随机化范围时，都应该保留配置版本和评估结果。

## 八、最重要的结论

当前最值得精读的不是几十篇论文，而是下面四类资料：

1. **Awesome Wheel-Legged Robots**：建立领域地图；
2. **CHAMP**：理解整机传统控制架构；
3. **Go2W RL Gym**：理解轮式四足从训练到部署的完整链路；
4. **Keep Rollin’**：理解轮子加入四足后，控制问题发生了什么变化。

当前代码的正确目标，不是立即让机器人“学会走”，而是先建立：

```text
统一 RobotState / RobotCommand 接口
+
有限状态机
+
单电机 / 单腿 / 单轮 / 站立 / 行驶测试
+
数据记录与安全保护
```

完成这些基础后，再阅读 RL 代码，才能知道每一段代码位于系统的哪一层、为什么需要它，以及它在仿真和真机之间如何传递。

最终应坚持下面的验证顺序：

```text
模型正确
  ↓
确定性控制正确
  ↓
仿真内策略正确
  ↓
跨仿真器验证正确
  ↓
安全限制正确
  ↓
低风险真机测试
  ↓
逐步扩大任务范围
```
