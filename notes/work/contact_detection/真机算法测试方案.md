# G1 真机接触观测算法部署与测试方案

本文档针对仓库中的自研接触观测算法：

```text
gear_sonic_deploy/src/g1/g1_deploy_onnx_ref/contact_observation/
```

目标是在 Unitree G1 板载 Orin（aarch64 / Ubuntu 22.04）上，**不启动
SONIC、不发布电机命令**，独立验证以下数据链路：

```text
G1 DDS
  ├── rt/lowstate：29 个电机 + 主 IMU
  ├── rt/secondary_imu：躯干 IMU
  └── rt/lowcmd：控制命令（可选）
              ↓
      DdsInputCollector
              ↓
      InputSynchronizer（50 Hz）
              ↓
      G1PinocchioDynamics
              ↓
      双臂关节力矩残差
              ↓
      DualArmWrenchEstimator
              ↓
  左右手腕局部坐标系六维外力
      [Fx,Fy,Fz,Mx,My,Mz]
```

测试必须遵循“先软件、后数据；先只读、后静态；先无接触、后已知负载；先吊架、
后落地”的顺序。任何阶段失败，都停留在当前阶段排查，不跳到下一阶段。

## 0. 先确定你要完成到哪一级

本方案把“配置完成”分成三级。必须先确定目标，否则很容易把
“四个 reader 都能运行”误当成“动态接触观测已经可用”。

| 完成等级 | 必须通过 | 完成后可以做什么 | 仍然不能做什么 |
| --- | --- | --- | --- |
| L1：真机只读链路配置完成 | 准备项 + G0–G3B | 确认 ARM64 环境、DDS、同步、URDF 和 Pinocchio 能正常运行 | 不能宣称 wrench 准确，不能用于接触决策 |
| L2：静态接触观测配置完成 | 准备项 + G0–G7，并完成第 18 节阈值制定 | 在已验证的静止/极低速条件下做诊断和候选接触判断 | 不能用于行走、躯干晃动、快速摆臂或 RL 控制 |
| L3：动态浮动基座配置完成 | 准备项 + G0–G8 全部通过 | 才可以进入 RL observation 候选和后续控制安全评审 | 仍不等于可直接控制真机，还需系统级安全验收 |

### 0.1 四个执行文件分别证明什么

```text
contact_input_reader
  只证明 DDS 原始数据能连续读取

contact_sync_reader
  只证明数据能按 50 Hz 因果同步

contact_dynamics_reader
  只证明 URDF/Pinocchio/关节残差链路能运行

contact_wrench_reader
  只证明残差能转换成左右腕 wrench，并能做质量检查
```

四个程序各运行一次不等于配置完成。完整验收还包括单元测试、
连续运行、多姿态基线、已知负载、外部测力和阈值标定。

### 0.2 现场唯一执行顺序

下表是本文档的主流程。每完成一行，立即在“状态”中打勾，并保存该步
日志。未满足“通过标准”时不得进入下一步。

| 状态 | 关卡 | 当前操作 | 通过后证明 | 通过后距离目标还差什么 |
| --- | --- | --- | --- | --- |
| [ ] | 安全准备 | 按第 4 节完成吊架/保护绳、急停、3 米清场、安全员和只读确认 | 具备开始真机测试的安全前提 | 还差软件和全部算法关卡 |
| [ ] | G0 | 按第 5–8 节完成 ARM64/Pinocchio/程序编译，运行所有指定单元测试 | 软件计算逻辑通过 | 还差真机 DDS、同步、动力学和 wrench 实测 |
| [ ] | 真机会话准备 | 按第 9–10 节创建日志目录、保存版本信息并确认 `$ROBOT_IF` | 本次测试可追溯，DDS 网卡已确定 | 还差 G1–G8 实际验证 |
| [ ] | G1 | 运行 `contact_input_reader` 30–60 秒 | 真机原始输入正常 | 还差时间同步和后续全部算法验证 |
| [ ] | G2 | 运行 `contact_sync_reader` 60 秒 | 50 Hz 快照和时间对齐正常 | 还差动力学、wrench、标定和动态基座 |
| [ ] | G3A | 不加 `--fixed-base` 运行 `contact_dynamics_reader` | 不完整浮动基座会被正确标记 `reliable=no` | 还差静态 fixed-base 动力学验证 |
| [ ] | G3B | 静止时加 `--fixed-base` 运行 60 秒 | 静态 URDF、关节映射和残差有效；此时 L1 完成 | 达到 L2 还差 G4–G7 和阈值制定 |
| [ ] | G4 | `contact_wrench_reader --fixed-base` 做单姿态无接触基线 | 静态 wrench 数值、可观性和质量标志正常 | 还差多姿态、已知负载、外部测力和阈值 |
| [ ] | G5 | 完成所有规定静态姿态的无接触测试 | 已知姿态相关偏置和病态区域 | 还差定量负载、外部真值和阈值 |
| [ ] | G6 | 左右手分别完成至少 3 个已知负载点 | 静态力的方向、单调性、线性和回零基本正确 | 还差外部测力真值、阈值和最终误差评估 |
| [ ] | G7 | 左右手使用外部测力设备完成 5/10/20 N 重复测试 | 静态外力相对于外部真值可量化 | 达到 L2 还差第 18 节阈值、迟滞和 unknown 逻辑 |
| [ ] | 阈值验收 | 按第 18 节固定阈值、连续帧和质量字段 | 此时 L2 静态接触观测配置完成 | 若目标是动态/RL，还差完整浮动基座 G8 |
| [ ] | G8 | 接入完整 `FloatingBaseKinematics`，通过回放、吊架动态和落地测试 | 动态阶段可持续得到 `reliable=yes` | 此时 L3 完成，才能申请接入 RL observation |

### 0.3 每一关的统一操作规则

每一关都必须按下面的方法执行：

1. 执行前记录机器人状态、命令和日志文件名；
2. 严格按当前关卡命令运行，不临时添加绕过质量检查的参数；
3. 对照该节“通过条件”逐项打勾；
4. 任何一项不通过，本关记为 `FAIL`，不继续下一关；
5. 全部通过后本关记为 `PASS`，并在第 22 节结论模板中更新当前完成等级。

一个关卡的命令“能启动”不等于该关通过；必须同时满足运行时长、数值、
质量标志和日志要求。

## 1. 本方案验证什么

### 1.1 验证目标

1. Pinocchio 2.7.0 能在 aarch64 上独立编译、链接并加载 G1 URDF；
2. Unitree SDK2 能从真机持续读取 LowState、主 IMU、躯干 IMU和 LowCmd；
3. 50 Hz 同步器能生成因果一致、时间误差受控的快照；
4. 29 个硬件关节与 URDF 关节映射正确；
5. Pinocchio 的重力、科氏力、惯性力和 RNEA 计算数值有效；
6. 左右臂 7 维残差能转换为左右腕局部坐标系六维外力；
7. 无接触基线、已知重量和外部测力测试具有可重复性；
8. 明确算法在何种条件下可以作为 RL observation，何种条件下只能诊断。

### 1.2 当前阶段不做

- 不运行 `deploy.sh`；
- 不启动 SONIC policy 或 planner；
- 不由接触算法控制机器人；
- 不在第一次测试中让机器人行走、跑、跳或快速摆臂；
- 不把 `--fixed-base` 的动态输出直接用于行走接触判断；
- 不在没有外部参考测力的情况下宣称六维力绝对准确。

## 2. 当前代码的安全属性和已知限制

### 2.1 四个独立程序均为只读

本方案使用：

```text
contact_input_reader
contact_sync_reader
contact_dynamics_reader
contact_wrench_reader
```

它们只创建 Unitree DDS subscriber，读取并打印数据，不创建 LowCmd publisher，
不会主动修改电机命令。运行时使用 `Ctrl+C` 正常退出。

### 2.2 当前浮动基座路径的限制

`G1PinocchioDynamics` 的严格浮动基座计算需要：

```text
基座世界位置
基座四元数
基座世界线速度
基座局部角速度
基座世界线加速度
基座局部角加速度
```

当前 `DdsInputCollector + InputSynchronizer` 只提供主 IMU 四元数和角速度，没有
完整的基座平动状态与角加速度。独立 reader 调用的是：

```cpp
Compute(snapshot);
```

因此浮动基座模式即使数值可以算出，也会按设计标记：

```text
valid=yes
base_kinematics_complete=false
residual_reliable=no
```

这不是程序崩溃，而是算法主动拒绝把不完整基座状态产生的残差声明为可靠。

静止或极低速诊断可以使用：

```bash
--fixed-base
```

此时会把基座视为固定，适合验证 URDF、关节映射、静态重力补偿、零点和已知
载荷。但机器人行走、躯干晃动或快速运动时，缺失的基座惯性会被误认为外力。

### 2.3 当前动力学补偿参数默认为零

代码中以下参数默认全零：

```text
armature
viscous_damping
coulomb_friction
```

因此当前残差是未标定基线：

```text
tau_residual_raw = tau_est - tau_rnea
```

真实电机传动摩擦、减速器效率、零偏、结构柔性和 `tau_est` 误差都会进入残差。
第一次真机测试的目标是测出这些偏差，而不是假设它们不存在。

### 2.4 腕部 wrench 在局部坐标系中

输出：

```text
wrench_external_local = [Fx,Fy,Fz,Mx,My,Mz]
```

是 `left_wrist_yaw_link` 或 `right_wrist_yaw_link` 局部坐标系下的结果。手腕姿态
变化后，同一个世界重力会投影到不同的局部轴。比较已知重量时必须：

1. 保持腕部姿态一致；或
2. 使用腕部姿态把局部 wrench 旋转到统一世界坐标系后再比较。

本节是第 0.2 节主流程的安全约束摘要。

| 关卡 | 内容 | 机器人状态 | 通过后才能进入 |

## 4. 测试前安全准备

1. 第一次测试时机器人必须使用吊架或可靠保护绳；
2. 清空机器人周围至少 3 米区域；
3. 准备 Unitree 硬件急停，安排独立安全员；
4. 算法终端只运行只读 reader；
5. 如果需要改变姿态，必须由已经验证过的 Unitree 控制器完成；
6. 不要为了测试 observer 临时编写未经验证的 LowCmd 发布器；
7. 真机测试中绝不使用 `--disable-crc-check`；
8. 任何 NaN、方向突变、持续振荡、温度异常或 DDS 中断都立即停止测试；
9. 已知负载必须牢固固定，不能在机器人上方或人员附近悬挂松动物体；
10. 接触力从小到大逐级增加，不能直接推墙或施加大冲击。

## 6. 在 aarch64 真机上单独安装 Pinocchio 2.7.0

### 6.1 为什么必须重新构建

Pinocchio 源码是跨平台的，但编译生成的库不是。开发电脑中的：

```text
install-x86_64/
build-x86_64/
```

不能复制到 Orin 使用。Orin 必须生成：

```text
build-aarch64/
install-aarch64/
```

项目 CMake 已按架构自动选择：

```text
contact_observation/third_party/pinocchio/install-aarch64
```

版本固定为 2.7.0，不使用系统中可能存在的其他 Pinocchio，也不使用 Conda 或
Python 的 `pip install pin` 替代 C++ 库。

### 6.2 安装 Pinocchio 编译依赖

在 Orin 上执行：

```bash
sudo apt update
sudo apt install -y \
  build-essential \
  cmake \
  pkg-config \
  git \
  libeigen3-dev \
  libboost-all-dev \
  liburdfdom-dev \
  liburdfdom-headers-dev
```

当前算法关闭 Python 接口和碰撞支持，因此不需要 EigenPy、Python bindings、
HPP-FCL/Coal 或 mesh collision 依赖。

### 6.3 方案 A：Orin 可以访问 GitHub

定义路径：

```bash
export G1_REPO="$HOME/g1_sonic/GR00T-WholeBodyControl"
export CONTACT_DIR="$G1_REPO/gear_sonic_deploy/src/g1/g1_deploy_onnx_ref/contact_observation"
export PIN_ROOT="$CONTACT_DIR/third_party/pinocchio"
mkdir -p "$PIN_ROOT"
cd "$PIN_ROOT"
```

下载固定版本及其子模块：

```bash
git clone \
  --branch v2.7.0 \
  --depth 1 \
  https://github.com/stack-of-tasks/pinocchio.git \
  source
```

```bash
cd source
git submodule update --init --recursive
git describe --tags --always
```

必须看到 `v2.7.0`。不要克隆当前 `master` 或最新 Pinocchio 4.x；当前代码和
CMake 明确按 Pinocchio 2.7 构建。

### 6.4 方案 B：Orin 无法访问 GitHub

在能联网的开发电脑上只下载源码和子模块：

```bash
mkdir -p ~/pinocchio_source_transfer
cd ~/pinocchio_source_transfer
git clone \
  --branch v2.7.0 \
  --depth 1 \
  https://github.com/stack-of-tasks/pinocchio.git \
  source
cd source
git submodule update --init --recursive
cd ..
tar -czf pinocchio-v2.7.0-source.tar.gz source
```

复制到 Orin：

```bash
scp pinocchio-v2.7.0-source.tar.gz <G1用户名>@<G1-IP>:~/
```

在 Orin 上解压到项目期望的位置：

```bash
export G1_REPO="$HOME/g1_sonic/GR00T-WholeBodyControl"
export CONTACT_DIR="$G1_REPO/gear_sonic_deploy/src/g1/g1_deploy_onnx_ref/contact_observation"
export PIN_ROOT="$CONTACT_DIR/third_party/pinocchio"
mkdir -p "$PIN_ROOT"
tar -xzf ~/pinocchio-v2.7.0-source.tar.gz -C "$PIN_ROOT"
```

可以跨机器复制 **源码**，但不能复制 x86_64 的 `build-*` 和 `install-*`。

### 6.5 在 Orin 上配置、编译和安装

```bash
export G1_REPO="$HOME/g1_sonic/GR00T-WholeBodyControl"
export CONTACT_DIR="$G1_REPO/gear_sonic_deploy/src/g1/g1_deploy_onnx_ref/contact_observation"
export PIN_ROOT="$CONTACT_DIR/third_party/pinocchio"
export PIN_SOURCE="$PIN_ROOT/source"
export PIN_BUILD="$PIN_ROOT/build-aarch64"
export PIN_INSTALL="$PIN_ROOT/install-aarch64"
```

检查源码和架构：

```bash
uname -m
test -f "$PIN_SOURCE/CMakeLists.txt" && echo "Pinocchio source OK"
```

配置：

```bash
cmake -S "$PIN_SOURCE" -B "$PIN_BUILD" \
  -DCMAKE_BUILD_TYPE=Release \
  -DCMAKE_INSTALL_PREFIX="$PIN_INSTALL" \
  -DCMAKE_INSTALL_LIBDIR=lib \
  -DBUILD_PYTHON_INTERFACE=OFF \
  -DBUILD_WITH_COLLISION_SUPPORT=OFF \
  -DBUILD_WITH_URDF_SUPPORT=ON \
  -DBUILD_TESTING=OFF \
  -DBUILD_EXAMPLES=OFF
```

Orin 内存紧张时先使用两个编译线程：

```bash
cmake --build "$PIN_BUILD" -j2
cmake --install "$PIN_BUILD"
```

如果 `tegrastats` 显示资源充足，可改为 `-j4`；不要一开始直接使用全部核心。

验证安装：

```bash
test -f "$PIN_INSTALL/lib/cmake/pinocchio/pinocchioConfig.cmake" \
  && echo "Pinocchio CMake config OK"
find "$PIN_INSTALL/lib" -maxdepth 1 -name 'libpinocchio*.so*' -exec file {} \;
grep -R "PACKAGE_VERSION" \
  "$PIN_INSTALL/lib/cmake/pinocchio"/pinocchioConfigVersion.cmake
```

动态库必须显示 ARM/aarch64，版本必须属于 2.7.x。

### 6.6 Pinocchio 不建议全局写入 `.bashrc`

Pinocchio 是本项目私有依赖，CMake 已支持显式路径。测试终端临时设置即可：

```bash
export CMAKE_PREFIX_PATH="$PIN_INSTALL:${CMAKE_PREFIX_PATH:-}"
export LD_LIBRARY_PATH="$PIN_INSTALL/lib:${LD_LIBRARY_PATH:-}"
```

这样不会覆盖系统、ROS 2 或其他项目使用的 Pinocchio。

## 7. 编译自研算法和测试程序

### 7.1 使用独立 ARM64 构建目录

不要复用开发电脑的 `gear_sonic_deploy/build/`。在 Orin 上使用独立目录：

```bash
export G1_REPO="$HOME/g1_sonic/GR00T-WholeBodyControl"
export DEPLOY_DIR="$G1_REPO/gear_sonic_deploy"
export CONTACT_DIR="$DEPLOY_DIR/src/g1/g1_deploy_onnx_ref/contact_observation"
export PIN_INSTALL="$CONTACT_DIR/third_party/pinocchio/install-aarch64"
export CONTACT_BUILD="$DEPLOY_DIR/build-contact-aarch64"

cd "$DEPLOY_DIR"
source ~/.bashrc
source scripts/setup_env.sh
export HAS_ROS2=0
```

配置：

```bash
cmake -S "$DEPLOY_DIR" -B "$CONTACT_BUILD" \
  -DCMAKE_BUILD_TYPE=Release \
  -DCONTACT_PINOCCHIO_ROOT="$PIN_INSTALL"
```

首次 CMake 配置会通过 `FetchContent` 获取 GoogleTest 1.14.0，因此 Orin 需要能
访问 GitHub。如果现场完全离线，应提前配置 GitHub 代理或把 GoogleTest 1.14.0
源码复制到 Orin，并通过 CMake 的
`FETCHCONTENT_SOURCE_DIR_GOOGLETEST` 指定本地源码目录。

只编译算法需要的目标：

```bash
cmake --build "$CONTACT_BUILD" \
  --target \
    run_tests \
    contact_input_reader \
    contact_sync_reader \
    contact_dynamics_reader \
    contact_wrench_reader \
  -j2
```

## 16. G6：已知重量定量测试

### 16.1 测试原则

- 从 0.25 kg 或 0.5 kg 开始；
- 负载牢固安装，不允许摆动；
- baseline 和 loaded 必须使用相同关节姿态、腕部姿态和温度条件；
- 每次加载前后都采集至少 20 至 30 秒；
- 比较差值，不直接比较带零偏的绝对值。

理论静态力：

```text
F = m * 9.81
```

示例：

| 质量 | 理论重力 |
| ---: | ---: |
| 0.25 kg | 2.45 N |
| 0.50 kg | 4.91 N |
| 1.00 kg | 9.81 N |
| 2.00 kg | 19.62 N |

如果负载质心相对腕部存在力臂 `r`：

```text
M = r × F
```

### 16.2 采集方法

无负载 baseline：

```bash
./target/release/contact_wrench_reader \
  "$ROBOT_IF" \
  --fixed-base \
  --allow-missing-torso \
  --warmup-seconds 5 \
  --duration-seconds 30 \
  --print-hz 10 \
  2>&1 | tee "$TEST_SESSION/06_left_baseline.log"
```

安装已知负载后，在完全相同姿态再次运行：

```bash
./target/release/contact_wrench_reader \
  "$ROBOT_IF" \
  --fixed-base \
  --allow-missing-torso \
  --warmup-seconds 5 \
  --duration-seconds 30 \
  --print-hz 10 \
  2>&1 | tee "$TEST_SESSION/06_left_0500g.log"
```

依次对左手、右手测试多个重量。不要添加 `--expect-no-contact`。

计算：

```text
delta_wrench = mean(loaded_wrench) - mean(baseline_wrench)
```

### G6 初步通过条件

- 加载侧力变化方向正确；
- 未加载侧不出现同等幅度变化；
- 质量增大时估计力单调增大；
- 卸载后回到原 baseline 附近；
- 至少三个载荷点呈近似线性；
- 初始工程目标可设为线性拟合 `R² >= 0.95`；
- 完成零偏/比例标定后，静态力幅值相对误差初步目标可设为不超过 20%。

20% 是初始工程指标，不是代码保证值。应根据传感器、负载安装和应用要求调整。

## 18. 从无接触分布制定接触阈值

不要直接永久使用 10 N 或 15 N。对每个代表性姿态统计：

```text
无接触均值 μ
无接触标准差 σ
99% / 99.9% 分位数
最大 condition
fit_rms 分布
```

第一版阈值可以使用：

```text
contact_on  = μ + 4σ
contact_off = μ + 2σ
```

使用不同的接通/断开阈值形成迟滞，避免在阈值附近抖动。还应加入：

- 连续 N 帧超过阈值才判定接触；
- `valid=false` 时输出 unknown，而不是 no-contact；
- `residual_reliable=false` 时输出 unknown；
- `saturated=true`、`rank<6` 或 condition 超限时输出 unknown；
- 解除接触后连续 N 帧低于 `contact_off` 才释放状态。

生产 observation 不应只传六维力，至少同时传递：

```text
wrench[6]
valid
residual_reliable
saturated
rank
condition
fit_rms
contact_state / unknown_state
```

## 19. G8：动态测试前必须完成的代码工作

当前 `--fixed-base` 只能完成静止和低速验证。要在行走、躯干运动或快速摆臂中
使用，必须先接入可靠状态估计器并调用：

```cpp

```

需要提供：

```cpp

  orientation_wxyz;
  linear_velocity_world_mps;
  angular_velocity_base_radps;
  linear_acceleration_world_mps2;
  angular_acceleration_base_radps2;
}
```

动态测试前还应完成：

1. 对 `dq`、`ddq` 和 `tau_est` 的噪声进行频谱分析；
2. 选择有明确相位延迟的滤波方案；
3. 标定 armature、黏性摩擦、库仑摩擦和静态偏置；
4. 校验基座/IMU/URDF 坐标系和四元数顺序；
5. 给基座状态加入时间戳并与 LowState 做因果同步；
6. 检测状态估计丢帧和时间跳变；
7. 记录 50 Hz 完整原始数据与输出，支持离线复现；
8. 在仿真或回放数据上先验证完整动态链路；
9. 吊架下进行小幅动态测试；
10. 最后才进入落地行走测试。

只要 `residual_reliable=no`，就不能把 wrench 作为可靠的动态接触 observation。

```text
[ ] 动态阶段 residual_reliable=yes
[ ] 已完成摩擦、armature 和零偏标定
[ ] 已建立迟滞、连续帧和 unknown 状态逻辑
[ ] observation 同时包含 wrench 与质量字段
[ ] 原始输入和算法输出可记录、可离线回放
[ ] 吊架动态测试通过后才允许落地测试
```

### 20.5 最终完成判定

只能从下列结论中选择一个：

1. **未完成 L1**：准备项或 G0–G3B 任意一关未通过。当前仍属于安全、软件、DDS、
   同步或动力学配置问题，不应开始 wrench 性能评价。
2. **L1 已完成**：准备项和 G0–G3B 全部通过。四个 reader 所需的基础链路已
   配置好，但尚未证明静态 wrench 准确。
3. **L2 已完成**：准备项和 G0–G7 全部通过，并且第 18 节的阈值、迟滞、连续帧
   和 unknown 状态已固定且留档。可以宣称“静态接触观测配置完成”。
4. **L3 已完成**：在 L2 基础上，完整浮动基座状态已接入，回放、吊架
   动态和落地测试全部通过，且动态过程中持续满足
   `residual_reliable=yes`。此时才可宣称“动态浮动基座配置完成”。

如果只运行了四个 reader，但没有完成 G4–G7 的重复采集和定量真值对比，
最高只能判定为 L1，不得判定为 L2 或 L3。

## 21. 常见失败与处理

### 21.1 Pinocchio CMake config 找不到

错误类似：

```text
Pinocchio 2.7 is not installed at .../install-aarch64
```

检查：

```bash
echo "$PIN_INSTALL"
find "$PIN_INSTALL" -name pinocchioConfig.cmake -print
```

重新配置主项目时显式传入：

```bash
-DCONTACT_PINOCCHIO_ROOT="$PIN_INSTALL"
```

### 21.2 `Exec format error`

说明复制了 x86_64 二进制：

```bash
file target/release/contact_wrench_reader
find "$PIN_INSTALL/lib" -name 'libpinocchio*.so*' -exec file {} \;
```

必须在 Orin 上重新编译。

### 21.3 `libpinocchio.so` 找不到

```bash
export LD_LIBRARY_PATH="$PIN_INSTALL/lib:${LD_LIBRARY_PATH:-}"
ldd target/release/contact_wrench_reader | grep 'not found'
```

同时检查程序的 BUILD_RPATH 是否来自 `install-aarch64/lib`。

### 21.4 一直等待 DDS 数据

```bash
ip -br -4 addr
echo "$ROBOT_IF"
env | grep -E 'CYCLONEDDS|ROS_DOMAIN|RMW_IMPLEMENTATION'
```

确认传入的是接口名而不是 IP，并确保 Unitree CycloneDDS 库路径位于 ROS 2 DDS
库之前。

### 21.5 torso IMU 始终为 0

先确认当前固件是否发布 `rt/secondary_imu`。如果确实不发布，使用：

```bash
--allow-missing-torso
```

并在测试报告中记录。不要把“允许缺少”写成“该信号已经验证”。

### 21.6 `residual_reliable=no`

浮动基座 reader 在当前代码下出现该结果是预期保护行为。静止诊断使用
`--fixed-base`；动态应用必须接入完整 `FloatingBaseKinematics`，不能通过修改
质量阈值或强行忽略该标志绕过。

### 21.7 rank 小于 6 或 condition 很大

机器人接近腕部 Jacobian 奇异姿态。改变关节姿态重新测试，并把该姿态列入算法
不可观或降权区域。不要在数值病态时继续用六维 wrench 做接触真值。

### 21.8 无接触偏置很大

依次检查：

1. 机器人实际手部与 URDF 是否匹配；
2. 负载、线缆、手部附件是否纳入模型；
3. 关节索引和正负方向；
4. `tau_est` 单位和符号；
5. 基座/重力方向；
6. 电机摩擦和 armature；
7. `ddq` 噪声；
8. 温度漂移；
9. 结构柔性和传动回差。

先修正模型与标定，不要仅通过放宽接触阈值掩盖问题。
